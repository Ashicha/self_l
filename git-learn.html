What is Git?
Understanding the version control system at its core

Git is a distributed version control system created by Linus Torvalds in 2005. Unlike older systems like SVN, there's no single "master server" — every developer has a complete copy of the entire history on their machine.

GitHub is just a website that hosts Git repositories in the cloud. Git is the tool; GitHub is the platform. They're separate things — you can use Git without GitHub.

🗄️
Git (Local)
The version control software installed on your machine. Tracks all your changes locally.
☁️
GitHub (Remote)
A cloud hosting service for Git repos. Adds collaboration features like PRs and Issues.
🔀
Distributed Model
Every clone IS the repo. You can commit, branch, and history — completely offline.
💡
Why does this matter? Since every clone is complete, you can work offline, collaborate without a central server, and never lose all history if one server goes down.

📁
Working Directory
The actual files on your disk. This is where you code. Changes here are "untracked" or "modified".
📋
Staging Area
A draft of your next commit. You explicitly add files here. Lets you craft precise commits.
💾
Local Repository
Your .git folder. When you commit, snapshots go here. Permanent history on your machine.
☁️
Remote Repository
GitHub, GitLab, etc. You push to share with others. You pull to sync others' changes.

Commits are snapshots, not diffs. This is key. Git stores a complete snapshot of all tracked files at each commit — not a list of changes. It uses smart compression so storage stays efficient, but conceptually it's full snapshots.

Git Object Types
# Git stores 4 types of objects, all content-addressed by SHA-1 hash blob — a file's content (no name, no path) tree — a directory (maps names/paths to blobs/trees) commit — a snapshot (points to a tree + parent commit) tag — a named pointer to a commit (for releases) # Each object is identified by its SHA hash git cat-file -t a1b2c3d # → "commit" git cat-file -p a1b2c3d # → view commit contents

Setup & Configuration
The config that affects every repo on your machine

Git has three levels of config: system (all users), global (your user), and local (one repo). Local overrides global overrides system.

Essential Setup
# First thing to do on a new machine git config --global user.name "Your Name" git config --global user.email "you@example.com" # Set default branch name to 'main' (modern standard) git config --global init.defaultBranch main # Set your preferred editor (VSCode, vim, nano, etc.) git config --global core.editor "code --wait" # Auto handle line endings (Windows users: true, Mac/Linux: input) git config --global core.autocrlf input # View all global config git config --global --list
.gitignore — What NOT to track
# .gitignore patterns — never commit these # Dependencies node_modules/ vendor/ # Environment & secrets (CRITICAL!) .env .env.local *.key secrets.json # Build outputs dist/ build/ *.pyc __pycache__/ # Editor files .vscode/ .idea/ *.swp # OS junk .DS_Store Thumbs.db
🚨
Never commit secrets! If you accidentally push an API key or password, assume it's compromised — GitHub scans for common secret patterns. Rotate it immediately, then use git filter-branch or BFG Repo Cleaner to remove it from history.

Staging & Commits
The art of crafting meaningful commits
The Commit Workflow
# Check what's changed git status # overview of tracked/untracked changes git diff # unstaged changes (working dir vs staging) git diff --staged # staged changes (staging vs last commit) # Stage files git add filename.js # stage one file git add . # stage everything (careful!) git add -p # interactive: pick specific hunks/lines to stage # Commit git commit -m "feat: add user login endpoint" git commit --amend # edit the LAST commit (before pushing!) # View history git log # full log git log --oneline # condensed one-line per commit git log --graph --oneline --all # visual branch graph

Conventional Commits — the industry standard format for commit messages. When everyone uses this, you can auto-generate changelogs and make code reviews much cleaner.

Type When to use Example
feat New feature feat: add dark mode toggle
fix Bug fix fix: prevent crash on empty input
docs Documentation only docs: update README install steps
style Formatting, no logic change style: format with prettier
refactor Restructure, no behavior change refactor: extract auth to service
test Adding/fixing tests test: add unit tests for parser
chore Deps, build, CI changes chore: bump node to v20
BREAKING CHANGE API-breaking change feat!: rename userId to user_id
✨
Pro tip: git add -p is one of the most underused commands. It lets you stage only specific lines of a file — perfect when you've mixed two unrelated changes in one file and want clean, atomic commits.

Branching
Work on multiple things in parallel without breaking each other

A branch in Git is just a lightweight pointer to a commit. Creating a branch doesn't copy any files — it just creates a new pointer. This is why branching is nearly instant in Git.

HEAD is a special pointer that tells Git "you are here" — it points to the current branch you're on.

🌿 Branch Visualizer
Branch Commands
# Create and switch to a new branch git checkout -b feature/user-auth git switch -c feature/user-auth # modern syntax (Git 2.23+) # Switch between branches git checkout main git switch main # List branches git branch # local branches git branch -r # remote branches git branch -a # all branches # Delete branches git branch -d feature/done # safe delete (merged only) git branch -D feature/done # force delete (even if unmerged) git push origin --delete feature/done # delete remote branch # Rename a branch git branch -m old-name new-name
🌿
Branch naming conventions: Use feature/, fix/, chore/, release/ prefixes. Use kebab-case. Keep names short but descriptive: feature/oauth-google not feature/implementing-google-oauth-login-button.

Merging & Rebasing
Two ways to integrate work — and when to use each

Merge integrates two branches by creating a new merge commit. It preserves the complete history of when branches diverged and merged. The safest option for shared/public branches.

git merge
# Merge feature branch into main git checkout main git merge feature/user-auth # If there are conflicts: # 1. Git marks conflicting files with <<<<< ===== >>>>> # 2. Edit files to resolve conflicts manually # 3. Stage resolved files git add resolved-file.js # 4. Complete the merge git commit # Fast-forward merge (when no divergence exists) git merge --ff-only feature/small-fix # Always create merge commit (no fast-forward) git merge --no-ff feature/user-auth # Abort a merge in progress git merge --abort

Rebase rewrites history by replaying your commits on top of another branch. The result is a clean, linear history — as if you'd started the branch from the current tip of main.

git rebase
# Rebase your feature branch onto main git checkout feature/user-auth git rebase main # If conflicts arise during rebase: # Fix conflict, then: git add . git rebase --continue # continue to next commit git rebase --abort # cancel and return to original state # Interactive rebase — rewrite last N commits git rebase -i HEAD~3 # modify last 3 commits # In interactive mode you can: # pick — keep commit as is # reword — change commit message # squash — combine with previous commit # fixup — combine, discard this message # drop — delete the commit entirely
🚨
Golden Rule of Rebasing: NEVER rebase commits that have been pushed to a shared remote branch. Rebasing rewrites commit SHAs. If others have those commits, their history will diverge and you'll create a mess. Only rebase your local, unpushed commits.
Scenario Use Merge Use Rebase
Integrating a feature into main ✓ Yes Sometimes
Updating your feature branch with main's new commits Works ✓ Preferred (cleaner)
Cleaning up messy local commits before PR ✗ No ✓ Yes (interactive)
Public/shared branch ✓ Safe ✗ Dangerous
Preserving exact history of when things happened ✓ Yes ✗ Rewrites history
07
Remotes, Push & Pull
Syncing your local work with the cloud
Working with Remotes
# View remotes git remote -v # Add a remote (usually done once) git remote add origin https://github.com/user/repo.git # Clone a repository git clone https://github.com/user/repo.git git clone git@github.com:user/repo.git # SSH (recommended) # Push commits to remote git push origin main git push -u origin feature/my-branch # -u sets upstream tracking git push # shortcut once upstream is set # Fetch vs Pull git fetch # downloads changes but DOESN'T merge (safe) git pull # fetch + merge (= git fetch + git merge) git pull --rebase # fetch + rebase (cleaner history) # View remote branches after fetch git branch -r # Track a remote branch locally git checkout -t origin/feature/their-branch
🔑
SSH vs HTTPS: Use SSH keys for GitHub. Once set up, you never type passwords. Generate one with ssh-keygen -t ed25519 -C "your@email.com", then add the public key to GitHub Settings → SSH Keys.
💡
Prefer git fetch over git pull when you want to see what changed before merging. Fetch is always safe — it just downloads objects. git pull --rebase is a great default for personal branches.
08
Pull Requests & Code Review
The GitHub collaboration workflow

A Pull Request (PR) is GitHub's way of proposing that your branch gets merged into another. It's not a Git feature — it's a GitHub feature built on top of Git. PRs are where code review happens.

1️⃣
Fork or Branch
For open source: fork the repo. For team projects: create a branch from main.
2️⃣
Push Your Branch
Push your feature branch to GitHub remote.
3️⃣
Open a PR
On GitHub, click "Compare & pull request". Write a good description!
4️⃣
Review & Iterate
Teammates leave comments. You push more commits to address feedback.
5️⃣
Merge
Once approved, merge the PR. Choose Merge, Squash, or Rebase merge.
6️⃣
Delete Branch
Delete the feature branch after merge — keep the repo clean.

PR merge strategies:

Strategy Result Best for
Merge commit Preserves all commits + adds merge commit Full traceability, shared branches
Squash & merge All PR commits squashed into ONE commit on main Clean main history, small PRs
Rebase & merge Linear history, each commit individually on main Clean history, all commits meaningful
📝
Good PR descriptions include: What changed and why (not how — the code shows that), screenshots if UI changed, how to test, and any breaking changes or migration steps.
09
GitHub Actions (CI/CD)
Automate testing, builds, and deployments

GitHub Actions lets you automate workflows triggered by Git events (push, PR, tag). It's how you do CI/CD — running tests on every PR, deploying on merge to main, etc.

Workflows live in .github/workflows/ as YAML files.

.github/workflows/ci.yml — Basic CI Pipeline
name: CI on: push: branches: [ main ] pull_request: branches: [ main ] jobs: test: runs-on: ubuntu-latest steps: - name: Checkout code uses: actions/checkout@v4 - name: Setup Node.js uses: actions/setup-node@v4 with: node-version: '20' cache: 'npm' - name: Install dependencies run: npm ci - name: Run linter run: npm run lint - name: Run tests run: npm test - name: Build run: npm run build
💡
Key concepts: Workflows are YAML files. Jobs run in parallel by default. Steps run sequentially within a job. Actions are reusable steps from the marketplace. Secrets are stored encrypted in repo settings.

Undoing Mistakes
Git can almost always save you — if you know which command to use

This is the section you'll reference most. Git is remarkably recoverable — almost nothing is truly lost. The key is knowing which undo operation fits your situation.

The Undo Toolkit
━━━━ BEFORE STAGING ━━━━ # Discard changes to a file (PERMANENT — no undo) git checkout -- filename.js git restore filename.js # modern syntax ━━━━ AFTER STAGING, BEFORE COMMIT ━━━━ # Unstage a file (keeps your changes) git reset HEAD filename.js git restore --staged filename.js # modern syntax ━━━━ AFTER COMMIT (LOCAL) ━━━━ # Undo last commit, keep changes staged git reset --soft HEAD~1 # Undo last commit, keep changes unstaged git reset --mixed HEAD~1 # (default) # Undo last commit and DISCARD changes (dangerous!) git reset --hard HEAD~1 ━━━━ AFTER PUSH (SHARED) ━━━━ # Revert a commit (creates new "undo" commit — safe) git revert a1b2c3d # revert specific commit git revert HEAD # revert last commit ━━━━ EMERGENCY: RESTORE LOST COMMITS ━━━━ git reflog # see ALL recent HEAD movements git checkout HEAD@{2} # go back to 2 moves ago git reset --hard HEAD@{1} # restore lost commits
⚠️
Reset vs Revert: git reset rewrites history (only safe for local commits). git revert adds a new commit that undoes changes (safe for public history). Use reset locally, revert on shared branches.

git stash — temporarily shelve changes so you can switch context:

git stash
# Save current changes to stash git stash git stash push -m "WIP: half-done login form" # List stashes git stash list stash@{0}: WIP: half-done login form stash@{1}: on main: old stash # Apply latest stash (keeps it in stash list) git stash apply # Apply and remove from stash list git stash pop # Apply specific stash git stash apply stash@{1} # Drop a stash git stash drop stash@{0}

Git Internals
Under the hood — what actually happens

Understanding Git's internals makes you a better Git user. You stop memorizing commands and start reasoning about what they do.

The .git directory is the entire repository. Delete it and you lose all history. Here's what's inside:

.git directory structure
.git/ ├── HEAD ← "You are here" — points to current branch ├── config ← repo-level git config ├── index ← the staging area (binary file) ├── objects/ ← ALL content (blobs, trees, commits) │ ├── a1/ ← first 2 chars of SHA become folder │ │ └── b2c3d4... ← rest of SHA is filename │ ├── pack/ ← compressed objects for efficiency │ └── info/ ├── refs/ ← branch/tag pointers │ ├── heads/ ← local branches (each file = branch name) │ │ └── main ← contains the SHA of main's tip │ ├── remotes/ ← remote tracking branches │ └── tags/ └── COMMIT_EDITMSG ← last commit message
Exploring objects (try this!)
# See what HEAD points to cat .git/HEAD ref: refs/heads/main # See what main points to cat .git/refs/heads/main a1b2c3d4e5f6... # Inspect any object git cat-file -t a1b2c3d # type: commit git cat-file -p a1b2c3d # pretty-print contents # A commit object looks like: tree 7d3b8a... ← points to the root tree parent f2c1e9... ← previous commit author Jane Doe <j@x.com> 1700000000 +0000 committer Jane Doe <j@x.com> 1700000000 +0000 feat: add user login endpoint
🧠
Key insight: Branches are just files containing a SHA. That's why creating/deleting branches is instant — Git just creates or deletes a tiny text file. There's no copying of files, no complex operations.
